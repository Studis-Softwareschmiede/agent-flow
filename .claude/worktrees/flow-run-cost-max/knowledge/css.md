# Knowledge Pack: css

Expertise für modernes CSS. Geladen als Domäne bei UI-Projekten. Regel-IDs: `css/R<NN>`.

## Coder-Guidance
- `css/R01` — Design-Tokens als Custom Properties (`--color-…`, `--space-…`); keine Magic-Werte.
- `css/R02` — Mobile-first: `@media (min-width: …)` progressiv.
- `css/R03` — Persistente Animationen unter `@media (prefers-reduced-motion: reduce)` abschalten.
- `css/R04` — Für **komponenten-bezogene** Responsiveness `@container` (Size Container Queries) statt globaler `@media`-Breakpoints nutzen — die Komponente reagiert auf ihren Container, nicht auf den Viewport. Eltern brauchen `container-type: inline-size` (oder `container`). Seit **August 2025 Baseline „Widely available"** (newly available seit Feb 2023), also produktionsreif ohne Polyfill. Quelle: [web.dev — Baseline digest Aug 2025](https://web.dev/blog/baseline-digest-aug-2025) · [MDN — CSS container queries](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_Container_Queries).
- `css/R05` — Light/Dark-Farben mit `light-dark(<light>, <dark>)` in **einer** Deklaration statt doppelter `prefers-color-scheme`-Blöcke. **Voraussetzung:** `color-scheme: light dark` (i. d. R. auf `:root`) — ohne das greift `light-dark()` nicht. Baseline „Newly available" seit **Mai 2024**. Quelle: [MDN — `light-dark()`](https://developer.mozilla.org/en-US/docs/Web/CSS/color_value/light-dark).
- `css/R06` — `:has()` ist **nicht forgiving**: Wird `:has()` (oder ein Argument darin) nicht unterstützt, fällt der **gesamte** Selektor-Block aus. Für graceful degradation in `:is()`/`:where()` kapseln (z. B. `:where(h1:has(+ h2))`). Baseline „Newly available" seit **Dez 2023**. Quelle: [MDN — `:has()`](https://developer.mozilla.org/en-US/docs/Web/CSS/:has).
- `css/R07` — Natives CSS Nesting (ohne Build-Step, ohne Sass) ist Baseline „Newly available" seit **2023-12-11**, erwartet „Widely available" ab **2026-06-11**. Wichtigste Falle: `&` ist bei Pseudo-Klassen/Pseudo-Elementen und Compound-Selektoren **zwingend** (`.card { &:hover {} }` korrekt); bei Nachfahren-Selektoren optional (`.card { .title {} }` = `.card .title`). Kein String-Concatenation à la Sass (`.block { &__element {} }` ist **ungültig** — erzeugt keinen Typ-Selektor). Quelle: [MDN — CSS nesting](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_nesting) · [web-features-explorer — nesting](https://web-platform-dx.github.io/web-features-explorer/features/nesting/).
- `css/R08` — `color-mix(in <colorspace>, <color1> [pct], <color2> [pct])` mischt zwei Farben in einem gewählten Farbraum (z. B. `oklab` für perceptuell gleichmässige Resultate, `srgb` für klasssisches Alpha-Blending). Baseline **„Widely available" seit 2025-11-09** — produktionsreif ohne Fallback. Ersetzt preprocessor-Tints/-Shades durch reines CSS (`color-mix(in oklab, var(--color-primary) 80%, white)`). Quelle: [MDN — `color-mix()`](https://developer.mozilla.org/en-US/docs/Web/CSS/color_value/color-mix) · [web-features-explorer — color-mix](https://web-platform-dx.github.io/web-features-explorer/features/color-mix/).
- `css/R09` — `@starting-style` definiert den **Ausgangs-Zustand** für CSS-Transitions beim ersten Render eines Elements (Entry-Transition) oder wenn `display` von `none` wechselt. Ohne `@starting-style` werden Transitions beim ersten Paint **nicht ausgelöst**. Baseline „Newly available" seit **2024-08-06** (~86 % globale Abdeckung); als Progressive Enhancement einsetzbar (ältere Browser überspringen die Eintritts-Animation, Layout bleibt korrekt). Kaskaden-Falle: `@starting-style` ist ein **separater Kaskaden-Mechanismus für den before-change state** (CSS Transitions Level 2 §3.3) — kein Spezifitäts-Konkurrent zum Haupt-Ruleset. Die Falle ist reine **Source-Order**: `@starting-style` hat dieselbe Spezifität wie das Haupt-Ruleset, weshalb es im Stylesheet **danach** stehen muss — steht es davor, gewinnt das Haupt-Ruleset per Source-Order und die Entry-Transition bleibt aus. Quelle: [MDN — `@starting-style`](https://developer.mozilla.org/en-US/docs/Web/CSS/@starting-style) · [CSS Transitions Level 2 §3.3](https://drafts.csswg.org/css-transitions-2/#defining-before-change-style) · [web-features-explorer — starting-style](https://web-platform-dx.github.io/web-features-explorer/features/starting-style/).
- `css/R10` — **View Transitions** (same-document): `document.startViewTransition(() => updateDOM())` + `view-transition-name: <ident>` auf beteiligten Elementen ermöglicht gerenderte Übergänge ohne manuelles Snapshot-Handling. Baseline **„Newly available" seit 2025-10-14** (Chrome 111+, Edge 111+, Firefox 144+, Safari 18+). Zwei harte Fallen: (1) **Doppelter Name** — haben zwei gleichzeitig sichtbare Elemente denselben `view-transition-name`, schlägt `ViewTransition.ready` fehl und der Übergang wird **still übersprungen**; (2) **Reduced Motion** — View Transitions sind Animationen und müssen mit `@media (prefers-reduced-motion: reduce) { ::view-transition-old(*), ::view-transition-new(*) { animation: none; } }` deaktiviert werden (R03 gilt hier analog). Progressive Enhancement: alte Browser fallen auf sofortigen DOM-Swap zurück. Quelle: [MDN — View Transition API](https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API) · [web-features-explorer — view-transitions](https://web-platform-dx.github.io/web-features-explorer/features/view-transitions/) · [web.dev — same-document view transitions Baseline](https://web.dev/blog/same-document-view-transitions-are-now-baseline-newly-available).
- `css/R11` — `text-wrap: balance` (oder `text-wrap-style: balance`) für **kurze Mehrzeiler** (Headings, Captions, Blockquotes) eliminiert einzelne überhängende Wörter am Zeilenende, indem Zeilen gleichmässig aufgeteilt werden. Baseline **„Newly available" seit 2024-05-13**. Wichtige Grenzen: Chromium wendet `balance` nur bei **≤ 6 Zeilen**, Firefox bei **≤ 10 Zeilen** an — für längeren Fliesstext daher wirkungslos. `text-wrap-style: pretty` hat **keinen Baseline-Status** (Firefox fehlt, Stand 2026-06) — in Chromium 130+ und Safari 26 nutzbar, aber nicht produktiv ohne Firefox-Fallback (Progressive Enhancement). Quelle: [MDN — `text-wrap`](https://developer.mozilla.org/en-US/docs/Web/CSS/text-wrap) · [MDN — `text-wrap-style`](https://developer.mozilla.org/en-US/docs/Web/CSS/text-wrap-style) · [web-features-explorer — text-wrap-balance](https://web-platform-dx.github.io/web-features-explorer/features/text-wrap-balance/).
- `css/R12` — **CSS Subgrid** (`grid-template-columns: subgrid` / `grid-template-rows: subgrid`) lässt einen verschachtelten Grid-Container die Track-Grössen des Eltern-Grids erben, statt eigene zu definieren — ermöglicht tiefe Ausrichtung ohne fixed-Werte oder JS. Baseline **„Widely available" seit 2026-03-15** — produktionsreif ohne Polyfill. Harte Falle: Wenn **beide** Dimensionen auf `subgrid` gesetzt sind, werden **keine impliziten Tracks** erzeugt; überschüssige Items stapeln sich im letzten Track. Lösung: `grid-auto-rows` für implizite Zeilen beizubehalten wenn nötig. Gap wird vom Eltern-Grid geerbt, kann aber mit `row-gap`/`column-gap` überschrieben werden. Quelle: [MDN — Subgrid](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_grid_layout/Subgrid) · [web-features-explorer — subgrid](https://web-platform-dx.github.io/web-features-explorer/features/subgrid/).

## Reviewer-Checklist
- Werte außerhalb der Spacing-/Token-Skala → **Important**.
- Kontrast < WCAG (Body 4.5:1 / Large 3:1) — **berechnen**, nicht schätzen → **Critical**.
- Animationen ohne `prefers-reduced-motion`-Berücksichtigung → **Important**.
- `@container`-Query ohne `container-type`/`container` auf einem Vorfahren → Query ist wirkungslos → **Important** (`css/R04`).
- `light-dark()` verwendet, aber kein `color-scheme: light dark` gesetzt → Funktion greift nicht → **Important** (`css/R05`).
- `:has()` außerhalb von `:is()`/`:where()` in einem Selektor, der auch unsupporteten Browsern dienen muss → kompletter Block-Ausfall → **Important** (`css/R06`).
- Sass-style `&__element`-Concatenation in nativem CSS → ungültig (kein Type-Selektor-Prefix) → **Important** (`css/R07`).
- `color-mix()` mit `srgb` statt `oklab`/`oklch` für Tints/Shades → perceptuell ungleichmässige Ergebnisse — `oklab` bevorzugen → **Minor** (`css/R08`).
- `@starting-style`-Block vor dem Haupt-Ruleset → gleiche Spezifität, Haupt-Styles gewinnen per **Source-Order** (`@starting-style` ist separater before-change-state-Mechanismus, kein Spezifitäts-Konkurrent), Entry-Transition fehlt → **Important** (`css/R09`).
- `view-transition-name` auf zwei gleichzeitig sichtbaren Elementen mit demselben Wert → `ViewTransition.ready` schlägt fehl, Übergang wird still übersprungen → **Critical** (`css/R10`).
- View-Transition-Animationen ohne `prefers-reduced-motion`-Guard (`::view-transition-old(*)/::view-transition-new(*)`) → **Important** (`css/R10` + `css/R03`).
- `text-wrap-style: pretty` in Produktion ohne Firefox-Fallback → kein Baseline-Status (Firefox fehlt, Stand 2026-06); Chromium 130+ und Safari 26 unterstützen es, Firefox nicht → **Important** (`css/R11`).
- Subgrid (`grid-template-columns/rows: subgrid`) auf beiden Achsen ohne `grid-auto-rows` → überschüssige Items stapeln sich im letzten Track, kein implizites Wachstum → **Important** (`css/R12`).

## Test-Approach
- Visueller Smoke; Kontrast der geänderten Farbpaare berechnet.
- Container-Queries: Komponente isoliert in schmalem/breitem Container rendern (nicht nur Viewport resizen) und Layout-Wechsel verifizieren.
- `light-dark()`: in beiden `color-scheme`-Modi (Light/Dark) prüfen.
- CSS Nesting: in DevTools CSSOM verifizieren dass Selektoren korrekt aufgelöst werden (Compound-Selektoren mit `&`).
- `@starting-style`: Element aus DOM entfernen und wieder einfügen (oder `display: none` → sichtbar) — Entry-Transition muss ausgelöst werden.
- View Transitions: `document.startViewTransition()` auslösen; in DevTools Animations-Panel prüfen ob `::view-transition-old/new` erzeugt werden; doppelte `view-transition-name`-Werte → Animation fehlt komplett (kein Fehler in der Console ohne explizites Promise-Handling).
- `text-wrap: balance`: in schmalen Containern auf kurze Headings (≤6 Zeilen) anwenden und Zeilenbrüche visuell verifizieren; in Browser ohne Support (Fallback: normal wrapping) prüfen dass Layout intakt bleibt.
- Subgrid: Nested-Grid-Items in einem Eltern-Grid rendern und prüfen, dass Spalten/Zeilen mit dem Parent-Grid ausgerichtet sind (DevTools Grid-Overlay nutzen).
