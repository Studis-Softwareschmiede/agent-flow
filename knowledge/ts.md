# Knowledge Pack: ts

Expertise für TypeScript. Geladen bei `profile.language: ts`. Regel-IDs: `ts/R<NN>`.

> **Hinweis Multi-Lang (PR-K):** in Mono-Repos mit `language: [java, ts]` (oder ähnlich) lädt der Reviewer diesen Pack via Per-File-Dispatch für `*.ts`/`*.tsx`-Dateien — Floor-Pack (`security.md`) gilt dateiunabhängig. Siehe `docs/architecture/framework-build-subsystem.md` §3.

## Coder-Guidance
- `ts/R01` — **`strict: true` in `tsconfig.json` Pflicht.** Aktiviert alle strict-Flags (`strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`, `alwaysStrict` …). Ohne strict ist TypeScript faktisch nur ein Linter; bei Migration aus Legacy-Code: schrittweise via einzelne strict-Flags, aber Ziel = Voll-strict. Quelle: [TS Handbook — strict](https://www.typescriptlang.org/tsconfig/#strict)
- `ts/R02` — **Kein `any`.** `any` deaktiviert Type-Checking — bei externen/unbekannten Inputs `unknown` + Type-Guards (`typeof`, `instanceof`, Discriminator-Check) verwenden. Wenn `any` zwingend (z.B. JSON.parse-Output kurzzeitig), explizit kommentieren (`// any: external input, validated below`). Quelle: [TS Handbook — any vs unknown](https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#unknown)
- `ts/R03` — **Immutability via `readonly` + `as const`.** Pflicht-Properties als `readonly` deklarieren (Compile-Time-Garantie), Literal-Arrays/-Objekte mit `as const` einfrieren (Tuple-Inferenz + readonly). Verhindert versehentliche Mutation, hilft dem Compiler bei Type-Narrowing. Quelle: [TS Handbook — const assertions](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-3-4.html#const-assertions)
- `ts/R04` — **Discriminated Unions statt Magic-String-Discriminators.** State/Result/Event als Union mit einem `kind`/`type`-Discriminator modellieren — der Compiler narrowt dann pro Case (exhaustive-check via `never`-Default). Kein `if (status === 'loading') { ... } else if (status === 'success') { ... }` mit lose verteilten String-Literalen. Quelle: [TS Handbook — Discriminated Unions](https://www.typescriptlang.org/docs/handbook/2/narrowing.html#discriminated-unions)
- `ts/R05` — **`import type` für Type-only Imports.** Reine Typen via `import type { Foo } from './foo'` importieren — der Import wird beim Compile entfernt, kein Runtime-Side-Effect, besseres Tree-Shaking. `verbatimModuleSyntax: true` in `tsconfig.json` erzwingt das. Quelle: [TS Handbook — Type-Only Imports](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-3-8.html#type-only-imports-and-export)
- `ts/R06` — **Pfad-Aliase via `tsconfig.paths` statt tiefer Relativ-Imports.** `import { x } from '@/lib/foo'` schlägt `import { x } from '../../../lib/foo'` (refactor-stabil, lesbar). Bundler-Konfig muss die Aliase ebenfalls auflösen (vite/webpack/jest/vitest haben das eingebaut, Native-Node braucht `tsconfig-paths` oder Build-Resolve).
- `ts/R07` — **Framework-/Build-Pack laden:** ist `profile.frameworks` oder `profile.build` gesetzt, lade zusätzlich die entsprechenden Packs aus `knowledge/frameworks/` und `knowledge/build/` (siehe `docs/architecture/framework-build-subsystem.md` §3). Framework-spezifische Regeln (Angular/React/Vue) stehen NICHT in diesem TS-Pack, sondern in `frameworks/<id>-<major>.md`.
- `ts/R08` — **`NoInfer<T>` für kontrollierte Typ-Inferenz in Generics (stabil seit TS 5.4).** `NoInfer<T>` signalisiert dem Compiler, den umschlossenen Typ nicht als Inferenz-Kandidat zu verwenden. Einsatz: wenn ein Parameter den Typ-Parameter _lesen_ soll (Constraint), aber nicht _beeinflussen_ darf (z.B. `defaultColor?: NoInfer<C>` in einer `createStreetLight<C extends string>(colors: C[], defaultColor?: NoInfer<C>)`-Signatur). Ohne `NoInfer` würde TS `C` aus beiden Parametern inferieren und eine ungültige Übergabe still akzeptieren. Quelle: [TS 5.4 Release Notes — The `NoInfer` Utility Type](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-4.html#the-noinfer-utility-type)
- `ts/R09` — **Inferred Type Predicates in `.filter()` (stabil seit TS 5.5).** TS inferiert automatisch einen Type-Predicate (`x is T`) für Funktionen, die eine boolesche Bedingung auf ihren Parameter zurückgeben — ohne explizites `(x): x is T`-Annotation. Praktische Folge: `.filter(x => x !== undefined)` liefert jetzt `T[]` statt `(T | undefined)[]`. Achtung: Truthiness-Checks (`.filter(x => !!x)`) lösen _keinen_ Predicate aus, weil `false` dann sowohl `undefined` als auch `0` bedeuten könnte (keine „if-and-only-if"-Semantik) — expliziter Vergleich nötig. Quelle: [TS 5.5 Release Notes — Inferred Type Predicates](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-5.html#inferred-type-predicates)
- `ts/R10` — **TS 5.6 flaggt syntaktisch konstante Truthy-/Nullish-Checks als Fehler.** Ausdrücke, die der Compiler _syntaktisch_ als immer wahr oder immer falsch erkennt, sind jetzt Compiler-Fehler — z.B. `/regex/` ohne `.test()` in einem `if`, eine Arrow-Function als Bedingung (`if (x => 0)`), oder der rechte Operand von `??`, der nie erreicht werden kann. Ausnahmen: idiomatische Muster wie `while (true)` oder `if (true || debugMode())` bleiben erlaubt. Fällt nicht unter `strict`, sondern ist immer aktiv ab TS 5.6. Quelle: [TS 5.6 Release Notes — Disallowed Nullish and Truthy Checks](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-6.html#disallowed-nullish-and-truthy-checks)
- `ts/R11` — **`--erasableSyntaxOnly` Flag für Node.js-native Typ-Stripping (stabil seit TS 5.8).** Aktiviert einen Compiler-Fehler für TS-exklusive Konstrukte mit Runtime-Semantik: `enum`-Deklarationen, `namespace`/`module` mit Code, Parameter-Properties in Klassen, `import =`/`export =`-Syntax. Ohne dieses Flag lassen sich solche Konstrukte nicht durch simples Typ-Strippen (Node.js 23.6+ `--experimental-strip-types`, Deno, Bun) ausführen. Neue Projekte, die bundlerlos/nativer Typ-Stripping anstreben, sollten das Flag setzen; Enums ggf. durch `const enum`-Alternativen oder Union-Types ersetzen. Quelle: [TS 5.8 Release Notes — The `--erasableSyntaxOnly` Option](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-8.html#the---erasablesyntaxonly-option)
- `ts/R12` — **`--rewriteRelativeImportExtensions` für `.ts`-Extension-Imports (stabil seit TS 5.7).** Ermöglicht es, TypeScript-Dateien mit ihrer tatsächlichen `.ts`/`.tsx`/`.mts`/`.cts`-Extension zu importieren (`import * as foo from "./foo.ts"`), während der Compiler diese beim Erzeugen von JS-Output automatisch in `.js`/`.jsx`/`.mjs`/`.cjs` umschreibt. Nützlich für Native-Node-ESM-Workflows und bundlerlose Setups, die ohne Import-Rewriting auskommen müssen. Setzt `--allowImportingTsExtensions` voraus (oder `--noEmit`). Quelle: [TS 5.7 Release Notes — Path Rewriting for Relative Paths](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-7.html#path-rewriting-for-relative-paths)
- `ts/R13` — **`import defer * as ns` für verzögerte Modul-Auswertung (stabil seit TS 5.9).** Importiert ein Modul als Namespace ohne es sofort auszuführen — die Modul-Auswertung findet erst beim ersten Zugriff auf einen Export statt (`import defer * as feature from "./feature.js"`). Nur Namespace-Form (`* as`) erlaubt, kein Named-/Default-`import defer`. Erfordert `--module preserve` oder `esnext`. TypeScript transpiliert das Konstrukt nicht herunter — laufzeitunterstützte Plattform nötig (oder entsprechendes Polyfill). Nützlich für Startup-Performance in großen Anwendungen (Feature-Code wird on-demand geladen). Quelle: [TS 5.9 Release Notes — Support for `import defer`](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-5-9.html#support-for-import-defer)
- `ts/R14` — **TS 6.0 kehrt zentrale Compiler-Default-Werte um (Breaking Change).** Seit TypeScript 6.0: `strict` default jetzt `true` (deckt sich mit `ts/R01` — wer bewusst unstrict bleiben will, muss ab jetzt explizit `"strict": false` setzen); `module` default `esnext` (statt `commonjs`); `target` default auf die jeweils aktuelle ES-Jahres-Version (aktuell `es2025`, „floating target"); `types` default `[]` (statt automatisch **alle** `@types/*`-Pakete aus `node_modules` einzubinden) — Node-Projekte verlieren dadurch beim Upgrade still Ambient-Globals wie `process`/`Buffer`, falls nicht explizit `"types": ["node"]` gesetzt wird. Beim Upgrade eines Bestandsprojekts auf TS 6.0: `tsconfig.json` gegen diese neuen Defaults gegenprüfen, nicht unverändert übernehmen. Quelle: [TS 6.0 Release Notes — `strict` is now `true` by default](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-6-0.html) · [`types` now defaults to `[]`](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-6-0.html) · [`module`/`target` neue Defaults](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-6-0.html)
- `ts/R15` — **`--moduleResolution node` (a.k.a. `node10`) deprecated seit TS 6.0.** Migration zu `nodenext` (wer direkt auf Node.js zielt) oder `bundler` (wer einen Bundler oder Bun einsetzt) — klassische Node-Resolution nicht mehr neu einführen. Quelle: [TS 6.0 Release Notes — Deprecated: `--moduleResolution node`](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-6-0.html)
- `ts/R16` — **`Map`/`WeakMap`-„Upsert"-Methoden `getOrInsert`/`getOrInsertComputed` (stabil seit TS 6.0, ECMAScript-„upsert"-Proposal Stage 4).** Ersetzen das klassische Doppel-Lookup-Anti-Pattern (`if (!map.has(key)) map.set(key, computeDefault())`) durch eine atomare Operation; `getOrInsertComputed` für teure/lazy Defaults (Callback läuft nur bei Miss). In der `esnext`-Lib verfügbar — `lib`/`target` entsprechend setzen. Quelle: [TS 6.0 Release Notes — New Types for „upsert" Methods](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-6-0.html)

## Reviewer-Checklist
- `tsconfig.json` ohne `strict: true` (oder ohne alle strict-Flags einzeln) → **Important** (R01).
- `any`-Type im Code ohne Begründungs-Kommentar → **Important** (R02).
- `// @ts-ignore` / `// @ts-expect-error` ohne Begründungs-Kommentar → **Important** (Suppression ohne Audit-Trail).
- Secrets/API-Keys inline statt aus Env → **Critical**.
- String-Interpolation in DB-Queries (siehe `sql`-Packs) → **Critical**.
- Magic-String-Discriminator statt Discriminated Union (mehrere `if`-Vergleiche gegen Literal-Strings) → **Suggestion** (R04).
- Regulärer `import { Foo }` für rein-typische Verwendung (`Foo` taucht nur in Type-Positions auf) → **Suggestion** (R05).
- Tiefe Relativ-Imports (`../../../`) → **Suggestion** (R06, Refactor-Brüchigkeit).
- Unbehandelte Promise-Rejection / fehlender `await` bei async-Funktion → **Important** (TypeScript flaggt das mit `noUncheckedSideEffectImports`/`noImplicitReturns` nicht direkt — Reviewer-Auge nötig).
- `.filter(x => !!x)` wo `x` nullable ist (falscher Truthiness-Predicate statt explizitem `!== undefined`) → **Suggestion** (R09, ergibt `(T | undefined)[]` statt `T[]`).
- Regex-Literal in `if`/`while` ohne `.test()` (immer truthy) → **Important** (R10, Compiler-Fehler ab TS 5.6).
- `enum` oder Parameter-Properties in Klassen in Projekten, die `--erasableSyntaxOnly` anstreben → **Suggestion** (R11, verhindert nativen Typ-Stripping-Betrieb).
- `.ts`/`.tsx`-Extension in relativen Imports ohne `--rewriteRelativeImportExtensions` im Build → **Important** (R12, führt zu Runtime-Fehlern „cannot find module ./foo.ts").
- `import defer * as ns` mit `--module commonjs`/`node16`/`node18` → **Important** (R13, kein Downlevel-Emit — laufzeitunterstützte Plattform und `--module preserve`/`esnext` nötig).
- Bestandsprojekt-Upgrade auf TS 6.0 ohne Gegenprüfung der neuen Defaults (`strict`/`module`/`target`/`types`), insb. fehlende `"types": ["node"]` bei Node-Projekten (stiller Verlust von `process`/`Buffer` etc.) → **Important** (R14).
- `tsconfig.json` mit `"moduleResolution": "node"`/`"node10"` in einem auf TS 6.0 migrierten Projekt → **Important** (R15, deprecated — zu `nodenext`/`bundler` migrieren).
- Doppel-Lookup-Pattern (`if (!map.has(key)) map.set(...)`) in einem Zielumfeld mit `--target esnext`/`"lib": ["esnext"]` → **Suggestion** (R16, `getOrInsert`/`getOrInsertComputed` nutzen).
- Test für einen Cross-Repo-/Cross-Komponenten-Vertrag baut den Fixture als handgeschriebenen Mock, der die Annahme der eigenen Komponente spiegelt, statt vom echten Produzenten-Output abzuleiten → **Important** (R14): der Mock hat per Konstruktion die vom Konsumenten erwartete Form und bestätigt so den Bug statt ihn zu fangen. Fixture aus realer Produzenten-Ausgabe **plus** ein Integrations-/Contract-Test Produzent→Konsument verlangen. Typ-Sicherheit hilft hier nur, wenn der Producer-Typ **geteilt** ist — ein lokal getippter Mock (`const x: LocalShape = {...}`) reproduziert dieselbe Lücke.

## Test-Approach
- **Type-Check Pflicht:** `tsc --noEmit` muss grün — kein TS-Code geht durch, wenn der Compiler rot ist.
- **Unit-Tests:** Framework-/Build-Tool-spezifisch — `vitest` (Vite-Projekte), `jest` (Bestand-Setups), `node --test` (Pure-Node ab Node 20). Siehe `knowledge/build/<build>.md` Test-Approach und ggf. Framework-Pack.
- **Lint:** ESLint mit `@typescript-eslint/recommended` als Floor; `eslint-plugin-import` für Import-Ordnung; ggf. Prettier für Formatting (out-of-scope dieses Packs).
- **Cross-Repo-Vertrag gegen echten Produzenten-Output testen (R14):** An einer Naht zwischen zwei Komponenten/Repos den Fixture aus realem Produzenten-Output ableiten, nie einen Mock erfinden, der die Konsumenten-Erwartung spiegelt; zusätzlich ein Integrations-/Contract-Test Produzent→Konsument. Ein geteilter Producer-Typ macht die Naht compile-fest — ein lokal getippter Mock nicht. *Fabrik-Präzedenz: 2026-07-08 regression-define (`status`-Feld), 2026-07-17 Story-Detail (`tok`-Objekt → `[object Object]`).*

## Spec-Tagging
Trace-Tag je gedecktem Kriterium gemäss `docs/architecture/traceability-subsystem.md`.
- **Idiom (Vitest/Jest/node:test):** kanonisches Token im `it()`/`test()`-Titel — analog js-Pack, gilt durchgehend für `.ts`-Testdateien: `it('@trace user-login#AC1,AC3 — rejects empty password', () => { … })`. Bei `test.each` das Token in den gemeinsamen Beschreibungs-String einbetten: `it.each(cases)('@trace user-login#AC1 — rejects %s', …)`. Vitest: filterbar via `vitest run -t "@trace user-login#AC1"`.
- **Idiom (Playwright — e2e):** Titel-Token wie oben: `test('@trace user-login#AC1 — Login-Flow durchläuft', …)` (optional native `{ tag: [...] }`, maßgeblich bleibt das Titel-Token).
- **Extraktions-Rezept:** Test-Titel einsammeln (`vitest list --json`, Jest-AST oder `grep -RoE` über `**/*.spec.ts`/`**/*.test.ts`), dann Core-Regex `@trace\s+([a-z0-9][a-z0-9-]*)#((?:AC\d+|BR-\d+)(?:,(?:AC\d+|BR-\d+))*)`.
- **Fallback:** kanonisches Token in der Test-Description; Core-Regex.
