## Coder-Guidance
- `python/R01` вҖ” **Type-Hints an Г¶ffentlichen Funktionssignaturen (PEP 484).** Parameter- und RГјckgabetypen Г¶ffentlicher Funktionen/Methoden annotieren (gradual typing) вҖ” macht die Schnittstelle fГјr Type-Checker (mypy/pyright) und IDE prГјfbar, ohne Laufzeitverhalten zu ГӨndern. Quelle: [PEP 484 вҖ” Type Hints](https://peps.python.org/pep-0484/)
- `python/R02` вҖ” **Native Generic-Syntax (PEP 695, stabil seit 3.12) statt `TypeVar`+`Generic`-Boilerplate.** FГјr Code, der Python вүҘ3.12 voraussetzt: `class Stack[T]: ...`, `def first[T](items: list[T]) -> T: ...`, `type ListOrSet[T] = list[T] | set[T]` statt `from typing import TypeVar, Generic` + explizitem `Generic[T]`-Base. KГјrzer, keine globalen `TypeVar`-Deklarationen, Varianz wird automatisch inferiert. Quelle: [PEP 695 вҖ” Type Parameter Syntax](https://peps.python.org/pep-0695/)
- `python/R03` вҖ” **Typing-Erweiterungen aus 3.13: Default-Werte fГјr Type-Parameter (PEP 696) + `TypeIs` (PEP 742).** `TypeVar`/`ParamSpec`/`TypeVarTuple` akzeptieren jetzt Defaults (`class Foo[T = int]: ...`); `typing.TypeIs` ist die verlГӨsslichere Alternative zu `TypeGuard` fГјr Narrowing-Funktionen (symmetrisches Narrowing in beiden Zweigen). Beide Features sind stabil. Quelle: [What's New in Python 3.13](https://docs.python.org/3/whatsnew/3.13.html)
- `python/R04` вҖ” **Deferred Annotation Evaluation (PEP 649) ist ab 3.14 Default.** Annotationen werden nicht mehr bei Definition ausgewertet, sondern lazy вҖ” Forward-Refs brauchen ab 3.14 keine String-Quotes (`def f(x: NotYetDefined) -> None: ...` funktioniert direkt). FГјr Code, der weiterhin <3.14 unterstГјtzen muss: `from __future__ import annotations` (bereits deferred) oder String-Quotes beibehalten. Introspektion via neues `annotationlib`-Modul (`get_annotations(fn, format=Format.STRING)` etc.). Quelle: [What's New in Python 3.14 вҖ” PEP 649](https://docs.python.org/3/whatsnew/3.14.html)
- `python/R05` вҖ” **`asyncio.TaskGroup` (seit 3.11) fГјr strukturierte NebenlГӨufigkeit statt losem `create_task()`/`asyncio.gather()`.** `TaskGroup` bietet stГӨrkere Sicherheitsgarantien: schlГӨgt eine Task fehl, werden alle Geschwister-Tasks automatisch gecancelt (bei `gather()` NICHT der Fall вҖ” verwaiste Tasks laufen weiter). Mehrere Exceptions werden zu einer `ExceptionGroup` zusammengefasst. Quelle: [asyncio docs вҖ” Task Groups](https://docs.python.org/3/library/asyncio-task.html)
- `python/R06` вҖ” **Exception Groups + `except*` (PEP 654, seit 3.11) fГјr Multi-Error-Handling.** Wenn mehrere unabhГӨngige Exceptions gleichzeitig propagiert werden mГјssen (z.B. aus `TaskGroup`, Bulk-Operationen, Cleanup-Code mit eigenem Fehler neben dem User-Fehler): `ExceptionGroup`/`BaseExceptionGroup` werfen, mit `except*ValueError as eg:` selektiv behandeln statt einer einzelnen `try/except`-Kette. Quelle: [PEP 654 вҖ” Exception Groups and except*](https://peps.python.org/pep-0654/)
- `python/R07` вҖ” **`venv` Pflicht fГјr Dependency-Isolation; nie ins System-Python installieren.** Jedes Projekt bekommt ein eigenes virtuelles Environment (`python -m venv .venv`); AbhГӨngigkeiten NIE global/systemweit installieren. Managed-System-Pythons (Distro-Paketmanager) verweigern globale `pip install`s bewusst (PEP 668, `EXTERNALLY-MANAGED`-Marker) вҖ” das ist Absicht, kein Bug, und darf nicht mit `--break-system-packages` umgangen werden. Quelle: [PEP 668 вҖ” Marking Python base environments as "externally managed"](https://peps.python.org/pep-0668/)
- `python/R08` вҖ” **`pathlib` statt `os.path`-String-Verkettung.** Objektorientierte Pfad-API (`Path`) statt prozeduraler `str`/`bytes`-Funktionen aus `os.path` вҖ” `/`-Operator fГјr Joins, eingebaute I/O-Methoden (`.read_text()`, `.write_bytes()`, `.iterdir()`), bessere Cross-Platform-Normalisierung. FГјr neue Code-Pfade Standard, `os.path` nur bei Interop-Zwang (Legacy-APIs, die `str` erwarten вҖ” dann `str(path)`). Quelle: [pathlib docs](https://docs.python.org/3/library/pathlib.html)
- `python/R09` вҖ” **`secrets`-Modul fГјr Tokens/PasswГ¶rter, nie `random`.** `random` ist fГјr Simulation/Modeling gedacht, NICHT kryptographisch sicher; fГјr Security-relevante Zufallswerte (Passwort-Reset-Tokens, Session-IDs, API-Keys) ausschlieГҹlich `secrets` (`secrets.token_urlsafe()`, `secrets.choice()`). Secrets selbst NIE inline im Code вҖ” aus Env-Variablen (`os.environ`) oder einem Secret-Store laden, nie hartcodiert committen. Quelle: [secrets docs](https://docs.python.org/3/library/secrets.html)
- `python/R10` вҖ” **`subprocess`: kein `shell=True` mit dynamisch gebauten/ungetrusteten Strings.** `subprocess` ruft standardmГӨГҹig keine Shell auf вҖ” Argumente als Liste (`subprocess.run(["cmd", arg1, arg2])`) sind damit von Haus aus injection-sicher. Wird `shell=True` explizit gesetzt, liegt die Verantwortung fГјr korrektes Quoting von Whitespace/Metazeichen beim Aufrufer (Shell-Injection-Risiko bei ungetrusteten Inputs). Quelle: [subprocess docs вҖ” Security Considerations](https://docs.python.org/3/library/subprocess.html#security-considerations)
- `python/R11` вҖ” **DB-Queries immer parametrisiert (Platzhalter), nie String-Formatierung/f-strings.** Analog zu jeder SQL-fГӨhigen Sprache: `cur.execute("SELECT * FROM t WHERE id = ?", (id,))` statt `f"SELECT * FROM t WHERE id = {id}"` вҖ” String-Bau ist SQL-Injection-anfГӨllig, Platzhalter werden vom Treiber sicher gebunden. Gilt fГјr `sqlite3` und alle DB-API-2.0-konformen Treiber. Quelle: [sqlite3 docs вҖ” How to use placeholders](https://docs.python.org/3/library/sqlite3.html#how-to-use-placeholders-to-bind-values-in-sql-queries) вҖ” *вҖһbeware of using Python's string operations to assemble queries, as they are vulnerable to SQL injection attacks"*
- `python/R12` вҖ” **`ruff` als vereinheitlichtes Lint+Format-Tool.** Ersetzt `flake8` (+Plugins), `black`, `isort`, `pyupgrade` durch ein einziges Rust-Tool; Konfiguration Гјber `[tool.ruff]`/`[tool.ruff.lint]`/`[tool.ruff.format]` in `pyproject.toml`. `ruff check` fГјr Linting, `ruff format` fГјr Formatting вҖ” beide teilen dieselbe Konfigurationsbasis (z.B. `line-length`). Quelle: [Ruff docs](https://docs.astral.sh/ruff/) / [Ruff Configuration](https://docs.astral.sh/ruff/configuration/)
- `python/R13` вҖ” **PEP 8 Namens-/Layout-Konventionen als Floor.** `snake_case` fГјr Funktionen/Variablen, `CapWords` fГјr Klassen, `ALL_CAPS` fГјr Modul-Konstanten, Exception-Klassen mit `Error`-Suffix (wenn tatsГӨchlich ein Fehler); Imports gruppiert in drei BlГ¶cken (Standardlib вҶ’ Third-Party вҶ’ Lokal) mit Leerzeile dazwischen. ZeilenlГӨnge laut PEP 8 79 Zeichen вҖ” in der Praxis oft per Tool-Konfig auf 88 (ruff-/Black-Default) angehoben; projektweit einheitlich halten. Quelle: [PEP 8 вҖ” Style Guide for Python Code](https://peps.python.org/pep-0008/)
- `python/R14` вҖ” **`Decimal`-Geld/-Mengen-PrГӨzision ist mit runden/binГӨr-exakten Fixtures beweislos вҖ” und hat systematische Footguns.** Beim Rechnen, Runden, Formatieren oder Schreiben von `Decimal`-Geld/-Mengen: (a) `float`вҶ’`Decimal` nur Гјber `Decimal(str(x))`, nie `Decimal(float)` (Гјbernimmt die BinГӨrdarstellung des Floats statt seines Dezimalwerts вҖ” `Decimal(4.9) == Decimal("4.9000000000000000355вҖҰ")`); (b) an jeder Schreibgrenze zu einer fixed-scale-Spalte/einem DTO explizit quantisieren вҖ” **auch bei reinem Pass-Through und bei Гҳ-Mittelung** (Division/Mittelung liefert nicht-terminierende Quotienten wie `0.9666вҖҰ667`, die eine fixed-scale-DB still rundet, SQLite/Tests aber nicht); (c) String-Ausgabe Гјber `format(d.normalize(), "f")` gegen RГјckfall in Scientific-Notation bei winzigen Summen (`str(Decimal(a)+Decimal(b))` вҶ’ `"1.334E-15"`); (d) `Decimal ** exponent` mit extern/on-chain kontrolliertem Exponenten gegen `ArithmeticError`/`OverflowError` absichern вҖ” die schmalen `InvalidOperation`/`ValueError` fangen `decimal.Overflow`/`decimal.DivisionByZero` NICHT (MRO: `DecimalException вҶ’ ArithmeticError`). **BeweisfГјhrung (Kern):** Tests mit ausschliesslich runden/binГӨr-exakten Werten (`5`, `50`, `100`, `Decimal("50")`, `5.0`) maskieren jede dieser Fallen vollstГӨndig вҖ” mindestens einen Testfall mit einem nicht-runden, nicht binГӨr-exakten Wert (`4.9`, `33.33`, ein 1:3-MengenverhГӨltnis) pro Decimal-Geld/-Mengen-Funktion einbauen. Quelle: [decimal docs вҖ” FAQ/Notes](https://docs.python.org/3/library/decimal.html)

## Test-Approach
- **Testrunner:** `pytest` als Standard. Fixtures fГјr Setup/Teardown via `conftest.py` (projekt-/paketweit geteilt), bevorzugt als Yield-Fixtures (`setup` vor `yield`, `teardown` danach) statt `request.addfinalizer()`. Parametrisierte TestfГӨlle Гјber `@pytest.mark.parametrize` statt kopierter Testfunktionen. Quelle: [pytest вҖ” How to use fixtures](https://docs.pytest.org/en/stable/how-to/fixtures.html)
- **Lint + Format Gate:** `ruff check` (Lint) und `ruff format --check` (Format-Verifikation) mГјssen grГјn sein, bevor ein Build als bestanden gilt. Quelle: [Ruff docs](https://docs.astral.sh/ruff/)
- **Decimal-PrГӨzisions-Fixture (R14):** jede Testfunktion, die `Decimal`-Geld/-Mengen berechnet, rundet, formatiert oder in eine fixed-scale-Spalte schreibt, deckt mindestens einen nicht-runden, nicht binГӨr-exakten Wert ab (`4.9`, `33.33`, ein nicht-terminierender Quotient aus Division/Гҳ-Mittelung). Runde Werte (`5`/`50`/`100`/`Decimal("50")`/`5.0`) sind binГӨr-exakt bzw. bereits quantisiert und beweisen weder PrГӨzision noch Quantisierung noch Format-Robustheit.
- **Type-Check:** statischer Type-Checker (`mypy` oder `pyright`) Гјber die annotierten Signaturen (R01вҖ“R03) laufen lassen вҖ” kein Pack-eigenes PrimГӨrquellen-Zitat hierfГјr (weder mypy noch pyright sind PSF-/Astral-PrimГӨrquellen dieses Packs), aber etablierte ErgГӨnzung zu PEP-484-Type-Hints in der Praxis.
- **Security-Smoke:** Secret-Scan (siehe `knowledge/security.md` вҡ‘ Floor) + Dependency-Audit (`pip-audit` o.ГӨ.) als Teil des Test-Approach, analog anderer Sprach-Packs.

## Spec-Tagging
Trace-Tag je gedecktem Kriterium gemГӨss `docs/architecture/traceability-subsystem.md`.
- **Idiom (pytest):** kanonisches Token im Docstring der Testfunktion (pytest-Testnamen sind Funktionsnamen, kein freier String вҖ” daher Docstring statt Titel-String wie bei JS/TS): `def test_rejects_empty_password():\n    """@trace user-login#AC1,AC3 вҖ” rejects empty password"""\n    ...`. Bei `@pytest.mark.parametrize` das Token im gemeinsamen Test-Docstring belassen, pro Case ggf. per `pytest.param(..., id=...)` differenzieren.
- **Extraktions-Rezept:** Test-Docstrings einsammeln (`grep -RoE '"""@trace[^"]*"""' **/test_*.py` oder AST-Parse via `ast.get_docstring()`), dann Core-Regex `@trace\s+([a-z0-9][a-z0-9-]*)#((?:AC\d+|BR-\d+)(?:,(?:AC\d+|BR-\d+))*)`.
- **Fallback:** kanonisches Token im Docstring des umschlieГҹenden Testmoduls, falls Funktions-Docstrings nicht gepflegt werden; Core-Regex bleibt gleich.
