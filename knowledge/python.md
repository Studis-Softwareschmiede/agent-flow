## Coder-Guidance
- `python/R01` — **Type-Hints an öffentlichen Funktionssignaturen (PEP 484).** Parameter- und Rückgabetypen öffentlicher Funktionen/Methoden annotieren (gradual typing) — macht die Schnittstelle für Type-Checker (mypy/pyright) und IDE prüfbar, ohne Laufzeitverhalten zu ändern. Quelle: [PEP 484 — Type Hints](https://peps.python.org/pep-0484/)
- `python/R02` — **Native Generic-Syntax (PEP 695, stabil seit 3.12) statt `TypeVar`+`Generic`-Boilerplate.** Für Code, der Python ≥3.12 voraussetzt: `class Stack[T]: ...`, `def first[T](items: list[T]) -> T: ...`, `type ListOrSet[T] = list[T] | set[T]` statt `from typing import TypeVar, Generic` + explizitem `Generic[T]`-Base. Kürzer, keine globalen `TypeVar`-Deklarationen, Varianz wird automatisch inferiert. Quelle: [PEP 695 — Type Parameter Syntax](https://peps.python.org/pep-0695/)
- `python/R03` — **Typing-Erweiterungen aus 3.13: Default-Werte für Type-Parameter (PEP 696) + `TypeIs` (PEP 742).** `TypeVar`/`ParamSpec`/`TypeVarTuple` akzeptieren jetzt Defaults (`class Foo[T = int]: ...`); `typing.TypeIs` ist die verlässlichere Alternative zu `TypeGuard` für Narrowing-Funktionen (symmetrisches Narrowing in beiden Zweigen). Beide Features sind stabil. Quelle: [What's New in Python 3.13](https://docs.python.org/3/whatsnew/3.13.html)
- `python/R04` — **Deferred Annotation Evaluation (PEP 649) ist ab 3.14 Default.** Annotationen werden nicht mehr bei Definition ausgewertet, sondern lazy — Forward-Refs brauchen ab 3.14 keine String-Quotes (`def f(x: NotYetDefined) -> None: ...` funktioniert direkt). Für Code, der weiterhin <3.14 unterstützen muss: `from __future__ import annotations` (bereits deferred) oder String-Quotes beibehalten. Introspektion via neues `annotationlib`-Modul (`get_annotations(fn, format=Format.STRING)` etc.). Quelle: [What's New in Python 3.14 — PEP 649](https://docs.python.org/3/whatsnew/3.14.html)
- `python/R05` — **`asyncio.TaskGroup` (seit 3.11) für strukturierte Nebenläufigkeit statt losem `create_task()`/`asyncio.gather()`.** `TaskGroup` bietet stärkere Sicherheitsgarantien: schlägt eine Task fehl, werden alle Geschwister-Tasks automatisch gecancelt (bei `gather()` NICHT der Fall — verwaiste Tasks laufen weiter). Mehrere Exceptions werden zu einer `ExceptionGroup` zusammengefasst. Quelle: [asyncio docs — Task Groups](https://docs.python.org/3/library/asyncio-task.html)
- `python/R06` — **Exception Groups + `except*` (PEP 654, seit 3.11) für Multi-Error-Handling.** Wenn mehrere unabhängige Exceptions gleichzeitig propagiert werden müssen (z.B. aus `TaskGroup`, Bulk-Operationen, Cleanup-Code mit eigenem Fehler neben dem User-Fehler): `ExceptionGroup`/`BaseExceptionGroup` werfen, mit `except*ValueError as eg:` selektiv behandeln statt einer einzelnen `try/except`-Kette. Quelle: [PEP 654 — Exception Groups and except*](https://peps.python.org/pep-0654/)
- `python/R07` — **`venv` Pflicht für Dependency-Isolation; nie ins System-Python installieren.** Jedes Projekt bekommt ein eigenes virtuelles Environment (`python -m venv .venv`); Abhängigkeiten NIE global/systemweit installieren. Managed-System-Pythons (Distro-Paketmanager) verweigern globale `pip install`s bewusst (PEP 668, `EXTERNALLY-MANAGED`-Marker) — das ist Absicht, kein Bug, und darf nicht mit `--break-system-packages` umgangen werden. Quelle: [PEP 668 — Marking Python base environments as "externally managed"](https://peps.python.org/pep-0668/)
- `python/R08` — **`pathlib` statt `os.path`-String-Verkettung.** Objektorientierte Pfad-API (`Path`) statt prozeduraler `str`/`bytes`-Funktionen aus `os.path` — `/`-Operator für Joins, eingebaute I/O-Methoden (`.read_text()`, `.write_bytes()`, `.iterdir()`), bessere Cross-Platform-Normalisierung. Für neue Code-Pfade Standard, `os.path` nur bei Interop-Zwang (Legacy-APIs, die `str` erwarten — dann `str(path)`). Quelle: [pathlib docs](https://docs.python.org/3/library/pathlib.html)
- `python/R09` — **`secrets`-Modul für Tokens/Passwörter, nie `random`.** `random` ist für Simulation/Modeling gedacht, NICHT kryptographisch sicher; für Security-relevante Zufallswerte (Passwort-Reset-Tokens, Session-IDs, API-Keys) ausschließlich `secrets` (`secrets.token_urlsafe()`, `secrets.choice()`). Secrets selbst NIE inline im Code — aus Env-Variablen (`os.environ`) oder einem Secret-Store laden, nie hartcodiert committen. Quelle: [secrets docs](https://docs.python.org/3/library/secrets.html)
- `python/R10` — **`subprocess`: kein `shell=True` mit dynamisch gebauten/ungetrusteten Strings.** `subprocess` ruft standardmäßig keine Shell auf — Argumente als Liste (`subprocess.run(["cmd", arg1, arg2])`) sind damit von Haus aus injection-sicher. Wird `shell=True` explizit gesetzt, liegt die Verantwortung für korrektes Quoting von Whitespace/Metazeichen beim Aufrufer (Shell-Injection-Risiko bei ungetrusteten Inputs). Quelle: [subprocess docs — Security Considerations](https://docs.python.org/3/library/subprocess.html#security-considerations)
- `python/R11` — **DB-Queries immer parametrisiert (Platzhalter), nie String-Formatierung/f-strings.** Analog zu jeder SQL-fähigen Sprache: `cur.execute("SELECT * FROM t WHERE id = ?", (id,))` statt `f"SELECT * FROM t WHERE id = {id}"` — String-Bau ist SQL-Injection-anfällig, Platzhalter werden vom Treiber sicher gebunden. Gilt für `sqlite3` und alle DB-API-2.0-konformen Treiber. Quelle: [sqlite3 docs — How to use placeholders](https://docs.python.org/3/library/sqlite3.html#how-to-use-placeholders-to-bind-values-in-sql-queries) — *„beware of using Python's string operations to assemble queries, as they are vulnerable to SQL injection attacks"*
- `python/R12` — **`ruff` als vereinheitlichtes Lint+Format-Tool.** Ersetzt `flake8` (+Plugins), `black`, `isort`, `pyupgrade` durch ein einziges Rust-Tool; Konfiguration über `[tool.ruff]`/`[tool.ruff.lint]`/`[tool.ruff.format]` in `pyproject.toml`. `ruff check` für Linting, `ruff format` für Formatting — beide teilen dieselbe Konfigurationsbasis (z.B. `line-length`). Quelle: [Ruff docs](https://docs.astral.sh/ruff/) / [Ruff Configuration](https://docs.astral.sh/ruff/configuration/)
- `python/R13` — **PEP 8 Namens-/Layout-Konventionen als Floor.** `snake_case` für Funktionen/Variablen, `CapWords` für Klassen, `ALL_CAPS` für Modul-Konstanten, Exception-Klassen mit `Error`-Suffix (wenn tatsächlich ein Fehler); Imports gruppiert in drei Blöcken (Standardlib → Third-Party → Lokal) mit Leerzeile dazwischen. Zeilenlänge laut PEP 8 79 Zeichen — in der Praxis oft per Tool-Konfig auf 88 (ruff-/Black-Default) angehoben; projektweit einheitlich halten. Quelle: [PEP 8 — Style Guide for Python Code](https://peps.python.org/pep-0008/)

## Reviewer-Checklist
- Secrets/API-Keys/Tokens inline im Code statt aus Env/Secret-Store geladen → **Critical** (R09).
- SQL-Query per String-Formatierung/f-string/`%`-Interpolation statt Platzhalter gebaut → **Critical** (R11, SQL-Injection).
- `subprocess.run(..., shell=True)` mit dynamisch aus User-/externem Input zusammengesetztem Command-String ohne Quoting → **Critical** (R10, Shell-Injection).
- `random`-Modul für Passwort-/Token-/Session-ID-Generierung statt `secrets` → **Critical** (R09).
- Nackte `except:` oder `except Exception:` ohne Re-Raise/Logging (verschluckt Fehler still) → **Important**.
- Fehlende Type-Hints an öffentlichen Funktions-/Methodensignaturen in neuem Code → **Important** (R01).
- `asyncio.gather()`/lose `create_task()`-Aufrufe für Sibling-Tasks, die bei Fehlschlag eines Tasks gemeinsam abgebrochen werden sollten, statt `asyncio.TaskGroup` → **Important** (R05).
- `pip install` (oder Äquivalent) ohne aktives `venv` — Hinweis auf fehlende Dependency-Isolation → **Important** (R07).
- Alte `TypeVar`+`Generic`-Boilerplate in neuem Code, das ohnehin Python ≥3.12 voraussetzt, statt PEP-695-Syntax → **Suggestion** (R02).
- `os.path`-String-Verkettung (`os.path.join`, manuelle `+`-Konkatenation) statt `pathlib.Path` in neuem Code → **Suggestion** (R08).
- Kein `pyproject.toml`/`[tool.ruff]`-Konfig bzw. Codebase erkennbar ungelintet → **Suggestion** (R12).
- PEP-8-Namensverstöße (z.B. `CamelCase`-Funktionsname, Exception-Klasse ohne `Error`-Suffix) → **Suggestion** (R13).

## Test-Approach
- **Testrunner:** `pytest` als Standard. Fixtures für Setup/Teardown via `conftest.py` (projekt-/paketweit geteilt), bevorzugt als Yield-Fixtures (`setup` vor `yield`, `teardown` danach) statt `request.addfinalizer()`. Parametrisierte Testfälle über `@pytest.mark.parametrize` statt kopierter Testfunktionen. Quelle: [pytest — How to use fixtures](https://docs.pytest.org/en/stable/how-to/fixtures.html)
- **Lint + Format Gate:** `ruff check` (Lint) und `ruff format --check` (Format-Verifikation) müssen grün sein, bevor ein Build als bestanden gilt. Quelle: [Ruff docs](https://docs.astral.sh/ruff/)
- **Type-Check:** statischer Type-Checker (`mypy` oder `pyright`) über die annotierten Signaturen (R01–R03) laufen lassen — kein Pack-eigenes Primärquellen-Zitat hierfür (weder mypy noch pyright sind PSF-/Astral-Primärquellen dieses Packs), aber etablierte Ergänzung zu PEP-484-Type-Hints in der Praxis.
- **Security-Smoke:** Secret-Scan (siehe `knowledge/security.md` ⚑ Floor) + Dependency-Audit (`pip-audit` o.ä.) als Teil des Test-Approach, analog anderer Sprach-Packs.

## Spec-Tagging
Trace-Tag je gedecktem Kriterium gemäss `docs/architecture/traceability-subsystem.md`.
- **Idiom (pytest):** kanonisches Token im Docstring der Testfunktion (pytest-Testnamen sind Funktionsnamen, kein freier String — daher Docstring statt Titel-String wie bei JS/TS): `def test_rejects_empty_password():\n    """@trace user-login#AC1,AC3 — rejects empty password"""\n    ...`. Bei `@pytest.mark.parametrize` das Token im gemeinsamen Test-Docstring belassen, pro Case ggf. per `pytest.param(..., id=...)` differenzieren.
- **Extraktions-Rezept:** Test-Docstrings einsammeln (`grep -RoE '"""@trace[^"]*"""' **/test_*.py` oder AST-Parse via `ast.get_docstring()`), dann Core-Regex `@trace\s+([a-z0-9][a-z0-9-]*)#((?:AC\d+|BR-\d+)(?:,(?:AC\d+|BR-\d+))*)`.
- **Fallback:** kanonisches Token im Docstring des umschließenden Testmoduls, falls Funktions-Docstrings nicht gepflegt werden; Core-Regex bleibt gleich.
